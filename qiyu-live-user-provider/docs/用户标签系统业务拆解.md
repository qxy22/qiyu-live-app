# 用户标签系统业务拆解

## 1. 业务背景

用户标签系统用于给用户打上各种业务标记，例如：

- 是否 VIP 用户
- 是否老用户
- 是否主播
- 是否付费用户
- 是否活跃用户
- 是否高价值用户
- 是否风险用户
- 是否测试用户

这些标签可以用于很多业务场景：

- 推荐系统：给活跃用户推荐更多内容。
- 风控系统：识别风险用户。
- 运营系统：筛选 VIP 用户、付费用户。
- 直播业务：判断用户是否主播。
- AB 实验：给测试用户打开特殊逻辑。

如果每个标签都单独建一列，例如：

```sql
is_vip
is_old_user
is_anchor
is_pay_user
is_active_user
```

随着标签越来越多，表字段会越来越多，维护成本会很高。

所以这个项目使用了“位运算标签系统”。

核心思想是：

```text
用一个 BIGINT 字段的二进制位，表示多个标签状态。
```

一个 `BIGINT` 有 64 个二进制位。每一位都可以表示一个标签。

例如：

```text
第 1 位：VIP 用户
第 2 位：老用户
第 3 位：主播用户
第 4 位：付费用户
第 5 位：活跃用户
```

这样一个 `BIGINT` 字段最多可以表达 64 个标签。

项目中设计了三个字段：

```text
tag_info_01
tag_info_02
tag_info_03
```

所以理论上最多可以表达：

```text
64 * 3 = 192 个标签
```

## 2. 核心数据结构

### 2.1 用户标签表

文档里的用户标签表是：

```sql
CREATE TABLE `t_user_tag` (
  `user_id` bigint NOT NULL DEFAULT -1 COMMENT '用户id',
  `tag_info_01` bigint NOT NULL DEFAULT '0' COMMENT '标签记录字段',
  `tag_info_02` bigint NOT NULL DEFAULT '0' COMMENT '标签记录字段',
  `tag_info_03` bigint NOT NULL DEFAULT '0' COMMENT '标签记录字段',
  `create_time` datetime DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  `update_time` datetime DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
  PRIMARY KEY (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb3 COLLATE=utf8mb3_bin COMMENT='用户标签记录';
```

字段含义：

| 字段 | 作用 |
| --- | --- |
| `user_id` | 用户 ID，也是主键 |
| `tag_info_01` | 第一组标签位 |
| `tag_info_02` | 第二组标签位 |
| `tag_info_03` | 第三组标签位 |
| `create_time` | 创建时间 |
| `update_time` | 更新时间 |

### 2.2 为什么要有三个 tag_info 字段

一个 `BIGINT` 可以存 64 个标签。

如果只有一个字段：

```text
tag_info_01
```

最多只能表达 64 个标签。

如果以后标签超过 64 个，就不够用了。

所以项目提前设计三个字段：

```text
tag_info_01: 标签 1 到标签 64
tag_info_02: 标签 65 到标签 128
tag_info_03: 标签 129 到标签 192
```

这样扩展性更好。

## 3. 位运算基础

### 3.1 二进制位

二进制只有 `0` 和 `1`。

一个数字可以转换成二进制。

例如：

```text
1  = 00001
2  = 00010
4  = 00100
8  = 01000
16 = 10000
```

这些数字有一个共同特点：二进制里只有一个位是 `1`。

所以它们适合表示“一个独立标签”。

### 3.2 标签值为什么用 1、2、4、8、16

因为这些值刚好对应不同的二进制位：

| 标签值 | 二进制 | 说明 |
| --- | --- | --- |
| `1` | `00001` | 第 1 位 |
| `2` | `00010` | 第 2 位 |
| `4` | `00100` | 第 3 位 |
| `8` | `01000` | 第 4 位 |
| `16` | `10000` | 第 5 位 |

如果使用 `3` 表示一个标签，会有问题。

因为：

```text
3 = 00011
```

它同时占用了第 1 位和第 2 位。

所以一个单独标签必须是：

```text
1, 2, 4, 8, 16, 32, 64 ...
```

也就是：

```java
1L << n
```

## 4. 设置标签

设置标签使用“或运算”：

```text
|
```

规则是：

```text
两个二进制位中只要有一个是 1，结果就是 1。
```

例如用户当前标签值是：

```text
00001
```

表示用户已经有第 1 个标签。

现在要给用户设置第 5 个标签：

```text
10000
```

执行：

```text
00001
| 10000
= 10001
```

结果是：

```text
10001
```

表示用户同时拥有第 1 个标签和第 5 个标签。

项目中的 SQL 是：

```sql
update t_user_tag
set ${fieldName} = ${fieldName} | #{tag},
    update_time = now()
where user_id = #{userId}
```

如果设置 `UserTagsEnum.IS_VIP`，它对应：

```java
fieldName = "tag_info_01"
tag = 1
```

最终效果就是：

```sql
tag_info_01 = tag_info_01 | 1
```

### 4.1 设置标签为什么是幂等的

幂等的意思是：

```text
执行一次和执行多次，最终结果一样。
```

比如用户已经有 VIP 标签：

```text
tag_info_01 = 00001
```

再次设置 VIP：

```text
00001
| 00001
= 00001
```

结果不变。

所以重复设置同一个标签不会出错。

## 5. 取消标签

取消标签使用：

```text
& ~
```

含义是：

```text
先对标签取反，再和当前标签值做与运算。
```

例如用户当前标签值是：

```text
10001
```

表示用户有第 1 个标签和第 5 个标签。

现在要取消第 5 个标签：

```text
10000
```

先取反：

```text
~10000 = 01111
```

再与运算：

```text
10001
& 01111
= 00001
```

最终只剩第 1 个标签。

项目中的 SQL 是：

```sql
update t_user_tag
set ${fieldName} = ${fieldName} & ~#{tag},
    update_time = now()
where user_id = #{userId}
```

如果取消 `UserTagsEnum.IS_VIP`，最终效果就是：

```sql
tag_info_01 = tag_info_01 & ~1
```

## 6. 判断是否包含标签

判断标签使用“与运算”：

```text
&
```

规则是：

```text
两个二进制位都为 1，结果才是 1。
```

例如用户当前标签值：

```text
10001
```

判断是否有第 5 个标签：

```text
10001
& 10000
= 10000
```

结果等于要判断的标签值，所以用户包含该标签。

项目中的 Java 代码是：

```java
return (tagInfo & tag) == tag;
```

完整工具类位置：

```text
qiyu-live-user-provider/src/main/java/org/qiyu/live/user/provider/utils/TagInfoUtils.java
```

## 7. 枚举设计

标签定义在：

```text
qiyu-live-user-interface/src/main/java/org/qiyu/live/user/constants/UserTagsEnum.java
```

示例：

```java
IS_VIP(UserTagFieldNameConstants.TAG_INFO_01, 1L, "VIP user"),
IS_OLD_USER(UserTagFieldNameConstants.TAG_INFO_01, 1L << 1, "old user"),
IS_ANCHOR(UserTagFieldNameConstants.TAG_INFO_01, 1L << 2, "anchor user"),
IS_PAY_USER(UserTagFieldNameConstants.TAG_INFO_01, 1L << 3, "paid user"),
IS_ACTIVE_USER(UserTagFieldNameConstants.TAG_INFO_01, 1L << 4, "active user"),
IS_HIGH_VALUE_USER(UserTagFieldNameConstants.TAG_INFO_02, 1L, "high value user"),
IS_RISK_USER(UserTagFieldNameConstants.TAG_INFO_02, 1L << 1, "risk user"),
IS_TEST_USER(UserTagFieldNameConstants.TAG_INFO_03, 1L, "test user");
```

每个枚举有三个属性：

| 属性 | 说明 |
| --- | --- |
| `fieldName` | 标签存在哪个字段，例如 `tag_info_01` |
| `tag` | 标签对应的二进制位，例如 `1L << 4` |
| `desc` | 标签描述 |

为什么要用枚举？

因为如果调用方直接传字段名和数字，容易传错。

例如：

```java
setTag(10001L, "tag_info_01", 16L)
```

这种写法不安全。

使用枚举后：

```java
setTag(10001L, UserTagsEnum.IS_ACTIVE_USER)
```

更清晰，也更不容易出错。

## 8. 常量设计

字段名定义在：

```text
qiyu-live-user-interface/src/main/java/org/qiyu/live/user/constants/UserTagFieldNameConstants.java
```

内容：

```java
public static final String TAG_INFO_01 = "tag_info_01";
public static final String TAG_INFO_02 = "tag_info_02";
public static final String TAG_INFO_03 = "tag_info_03";
```

为什么字段名要做成常量？

因为 Mapper 里使用了动态字段：

```java
${fieldName}
```

动态字段不能用普通的 `#{}` 参数绑定。

为了避免传入非法字段名，业务层只允许从 `UserTagsEnum` 中取字段名，而枚举又只使用固定常量。

这样可以降低 SQL 注入风险。

```text

```

## 11. Service 层

接口位置：

```text
qiyu-live-user-provider/src/main/java/org/qiyu/live/user/provider/service/IUserTagService.java
```

实现位置：

```text
qiyu-live-user-provider/src/main/java/org/qiyu/live/user/provider/service/impl/UserTagServiceImpl.java
```

核心方法：

```java
public boolean setTag(Long userId, UserTagsEnum userTagsEnum) {
    if (!isValidRequest(userId, userTagsEnum)) {
        return false;
    }
    userTagMapper.initUserTag(userId);
    return userTagMapper.setTag(userId, userTagsEnum.getFieldName(), userTagsEnum.getTag()) > 0;
}
```

这段逻辑做了三件事：

1. 校验 `userId` 和 `userTagsEnum` 是否合法。
2. 如果用户标签记录不存在，先初始化一条。
3. 调用 Mapper 执行位运算更新。

为什么要先初始化？

因为如果用户在 `t_user_tag` 中没有记录，直接执行：

```sql
update t_user_tag set ...
where user_id = ?
```

会更新 0 行。

所以先执行：

```sql
INSERT IGNORE INTO t_user_tag ...
```

如果用户没有记录，就插入。

如果已经有记录，就忽略。

## 12. Mapper 层

Mapper 位置：

```text
qiyu-live-user-provider/src/main/java/org/qiyu/live/user/provider/dao/mapper/IUserTagMapper.java
```

### 12.1 初始化用户标签记录

```java
@Insert("""
        INSERT IGNORE INTO t_user_tag
        (user_id, tag_info_01, tag_info_02, tag_info_03, create_time, update_time)
        VALUES (#{userId}, 0, 0, 0, NOW(), NOW())
        """)
int initUserTag(@Param("userId") Long userId);
```

作用：

```text
用户没有标签记录时插入一条空记录。
```

为什么用 `INSERT IGNORE`？

因为 `user_id` 是主键。

如果记录已经存在，再插入会主键冲突。

`INSERT IGNORE` 可以忽略这个冲突。

### 12.2 设置标签

```java
@Update("update t_user_tag set ${fieldName} = ${fieldName} | #{tag}, update_time = now() where user_id = #{userId}")
int setTag(@Param("userId") Long userId, @Param("fieldName") String fieldName, @Param("tag") long tag);
```

执行效果：

```sql
update t_user_tag
set tag_info_01 = tag_info_01 | 16
where user_id = 10001;
```

### 12.3 取消标签

```java
@Update("update t_user_tag set ${fieldName} = ${fieldName} & ~#{tag}, update_time = now() where user_id = #{userId}")
int cancelTag(@Param("userId") Long userId, @Param("fieldName") String fieldName, @Param("tag") long tag);
```

执行效果：

```sql
update t_user_tag
set tag_info_01 = tag_info_01 & ~16
where user_id = 10001;
```

## 14. 分表设计

文档中要求创建 100 张用户标签表：

```text
t_user_tag_00
t_user_tag_01
t_user_tag_02
...
t_user_tag_99
```

为什么要分表？

用户标签是按用户维度存储的，如果用户量很大，单表数据会越来越多。

按照 `user_id % 100` 分表，可以把数据分散到 100 张表里。

例如：

```text
user_id = 10001
10001 % 100 = 1
落到 t_user_tag_01
```

```text
user_id = 10088
10088 % 100 = 88
落到 t_user_tag_88
```

项目中 ShardingSphere 配置位置：

```text
qiyu-live-user-provider/src/main/resources/qiyu-db-sharding.yaml
```

新增逻辑表：

```yaml
t_user_tag:
  actualDataNodes: readwrite_ds.t_user_tag_${(0..99).collect(){it.toString().padLeft(2,'0')}}
  tableStrategy:
    standard:
      shardingColumn: user_id
      shardingAlgorithmName: t_user_tag-inline
```

路由算法：

```yaml
t_user_tag-inline:
  type: INLINE
  props:
    algorithm-expression: t_user_tag_${(user_id % 100).toString().padLeft(2,'0')}
```

业务代码仍然写：

```sql
update t_user_tag ...
```

ShardingSphere 会自动路由到真实表：

```text
t_user_tag_01
```

## 15. 一次设置标签的完整链路

调用代码：

```java
userTagRpc.setTag(10001L, UserTagsEnum.IS_ACTIVE_USER);
```

完整流程：

```text
1. 调用方通过 Dubbo 调用 IUserTagRpc.setTag
2. 请求进入 UserTagRpcImpl.setTag
3. RPC 层调用 IUserTagService.setTag
4. Service 层校验 userId 和 userTagsEnum
5. Service 层调用 initUserTag 初始化用户标签记录
6. Mapper 执行 INSERT IGNORE INTO t_user_tag
7. ShardingSphere 根据 user_id % 100 路由到 t_user_tag_01
8. Service 层调用 Mapper.setTag
9. Mapper 执行 tag_info_01 = tag_info_01 | tag
10. 数据库保存新的标签值
```

假设 `IS_ACTIVE_USER` 是：

```java
1L << 4
```

也就是：

```text
10000
```

如果用户原来的 `tag_info_01` 是：

```text
00001
```

设置后变成：

```text
10001
```

## 16. 一次取消标签的完整链路

调用代码：

```java
userTagRpc.cancelTag(10001L, UserTagsEnum.IS_ACTIVE_USER);
```

完整流程：

```text
1. 调用方通过 Dubbo 调用 IUserTagRpc.cancelTag
2. 请求进入 UserTagRpcImpl.cancelTag
3. RPC 层调用 IUserTagService.cancelTag
4. Service 层校验参数
5. Service 层初始化用户标签记录
6. Mapper 执行 tag_info_01 = tag_info_01 & ~tag
7. 数据库保存取消后的标签值
```

如果用户原来的 `tag_info_01` 是：

```text
10001
```

取消 `10000` 后：

```text
00001
```

## 17. 一次判断标签的完整链路

调用代码：

```java
boolean result = userTagRpc.containTag(10001L, UserTagsEnum.IS_ACTIVE_USER);
```

完整流程：

```text
1. 调用方通过 Dubbo 调用 IUserTagRpc.containTag
2. 请求进入 UserTagRpcImpl.containTag
3. RPC 层调用 IUserTagService.containTag
4. Service 层根据 user_id 查询 UserTagPO
5. 根据 UserTagsEnum.fieldName 判断应该读取哪个字段
6. 调用 TagInfoUtils.isContain
7. 执行 (tagInfo & tag) == tag
8. 返回 true 或 false
```

## 19. 如何新增一个标签

假设要新增“黑名单用户”标签。

如果它属于 `tag_info_02`，并且使用第 3 位：

```java
IS_BLACK_USER(UserTagFieldNameConstants.TAG_INFO_02, 1L << 2, "black user");
```

注意：

同一个字段内，不能重复使用相同 bit。

错误示例：

```java
IS_RISK_USER(TAG_INFO_02, 1L << 1)
IS_BLACK_USER(TAG_INFO_02, 1L << 1)
```

这两个标签会共用同一个二进制位，无法区分。

## 20. 当前实现的优点

- 存储紧凑：一个 `BIGINT` 可存 64 个标签。
- 设置高效：设置标签只需要一次位运算更新。
- 取消高效：取消标签只需要一次位运算更新。
- 判断高效：判断标签只需要一次位运算。
- 可扩展：三个字段可支持 192 个标签。
- 支持分表：按 `user_id % 100` 路由到 100 张表。
- 调用清晰：通过 `UserTagsEnum` 避免调用方乱传字段名和 bit 值。

## 21. 当前实现需要注意的点

### 21.1 `${fieldName}` 的安全问题

Mapper 中使用了：

```java
${fieldName}
```

这是字符串直接拼接 SQL。

如果 `fieldName` 来自用户输入，会有 SQL 注入风险。

当前实现中，`fieldName` 只来自 `UserTagsEnum`，不是外部直接传入，所以风险可控。

### 21.2 分表必须提前创建

ShardingSphere 不会自动帮你建真实表。

如果执行：

```sql
update t_user_tag ...
```

但真实表 `t_user_tag_01` 不存在，就会报错。

所以启动前要先执行建表存储过程。

### 21.3 标签枚举要统一维护

新增标签时必须统一加到 `UserTagsEnum`。

不要在业务代码中直接写：

```java
16L
```

否则后续很难维护。

## 22. 学习重点总结

这个业务最核心的知识点有五个：

1. 用二进制位表示标签。
2. 用 `|` 设置标签。
3. 用 `& ~` 取消标签。
4. 用 `&` 判断标签。
5. 用分表解决用户维度数据量增长问题。

如果你能理解下面三行，就理解了标签系统的核心：

```sql
tag_info_01 = tag_info_01 | tag
tag_info_01 = tag_info_01 & ~tag
```

```java
(tagInfo & tag) == tag
```
